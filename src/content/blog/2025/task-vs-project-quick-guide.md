# 📌 Quick Guide: Task vs. Project

## **Task**

* **Definition:** A single, concrete action you can complete in one work session.
* **Typical duration:** Minutes to 2 hours.
* **Examples:**

  * Call a client.
  * Fix a specific bug.
  * Send a follow-up email.
* **Rule:** If it fits in one sentence starting with a verb → it’s a task.

---

## **Project**

* **Definition:** A set of tasks leading to a larger outcome.
* **Typical duration:** Days, weeks, or months.
* **Examples:**

  * Implement a new chatbot.
  * Prepare a full demo.
  * Migrate a reporting system.
* **Rule:** If it needs multiple steps, coordination, or intermediate deliverables → it’s a project.

---

## **How to Distinguish Quickly**

1. Can it be solved in **one action**?

   * Yes → **Task**
   * No → **Project**

2. Does it require **multiple steps or people**?

   * Yes → **Project**
   * No → **Task**

3. Does it take **more than 2 hours or 2 steps**?

   * Yes → **Project**

---

## **The Camel Task Trap 🐪**

* Looks like a task, but drags on for days.
* Example: *“Implement OAuth”* → actually: read docs, generate keys, test, debug.
* Result: nothing gets ticked off → you feel stuck.

### **How to Avoid It**

* Break it down into subtasks once it starts dragging.
* Set a **max task duration (2–3h)**. If it exceeds → make it a project.
* Define clear deliverables: not *“set up OAuth”*, but *“OAuth returns valid token in Postman”*.

---

## **Why It Matters**

* **Motivation:** Crossing things off keeps momentum.
* **Clarity:** Backlog stays realistic.
* **Focus:** Daily wins add up while keeping the big picture.

---

👉 Golden rule: **“If it lasts more than 2 hours or has more than 2 steps, it’s a project.”**
